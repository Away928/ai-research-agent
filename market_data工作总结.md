## 一、架构设计

参考 Wind 金融终端的思路，将**个股行情**和**行业板块行情**拆成两个独立模块，各自从不同在线数据源获取，互不依赖。

```
个股行情                        行业行情
├─ 腾讯财经 (qt.gtimg.cn)        ├─ 新浪指数 (sz399363 国证算力)
├─ 新浪财经 (hq.sinajs.cn)       ├─ 成分股PE中位数推导
└─ baostock (HTTP日线)           └─ 成分股市值加总推导
       │                                │
       ▼                                ▼
  逐字段选最优源                   独立获取 + 汇总推导
       │                                │
       └──────────┬─────────────────────┘
                  ▼
            缓存 (industry_data.json / financial_data.json / news_cache.json, 7天有效)
                  │
                  ▼
           无可用数据 → 诚实返回空，不做假兜底
```

### 核心原则

**逐字段、逐源询问"你有没有这个数据"，有的取精度最高的，没有的降级。不因为某个源挂了就整体崩溃。不再用硬编码种子数据冒充真实数据。**

## 二、数据源覆盖总表

### 三个在线数据源

| 数据源 | 协议 | 延迟 | 能提供的字段 |
|--------|------|------|-------------|

## 三、降级机制

```
在线源通 → 返回数据 + 自动写缓存
在线源全挂 → 读缓存（7天内有数据直接用，超过7天标注"过期"但仍然返回）
缓存也不可用 → 返回空/标注"无可用数据"（诚实，不做假兜底）
```

### 种子数据全面移除（2026-07-10）

过去在降级链末端有硬编码的种子数据（价格/PE/市值/财务/新闻），2026-07-10 全部移除。原因：
- 行情数据每天都在变，硬编码值毫无参考意义
- 财务指标（营收增速/ROE 等）估算值不可靠，会误导分析
- 新闻种子日期会越来越旧，对投研没有价值
- 不如诚实地说"当前无可用数据，请联网后重试"

唯一保留的"种子"是成分股列表的 `code + name`，因为这些是稳定的标识信息，离线状态下至少能告诉用户"这个行业包含哪些公司"。

### 财务数据的降级（2026-07-10 重写）

```
baostock 日线（3个接口：profit/growth/dupont）→ 缓存(7天) → 返回空
```

5 个指标：ROE、毛利率、营收增速、净利润增速、资产负债率。每个行业 15 只成分股全覆盖。

### 新闻的降级（2026-07-10 重写）

```
AKShare stock_news_em（3家龙头公司的新闻聚合去重）→ 缓存(7天) → 返回空
```

### 公告/研报的降级

```
巨潮资讯网 cninfo.com.cn（公告）→ 缓存(7天) → 返回空
研报暂无免费在线源，返回空
```

## 五、关键设计决策

### 1. price 直接用腾讯，不做双源交叉验证
- 腾讯延迟 ~150ms vs 新浪 ~3s，精度明显更高。新浪仅作备份。

### 2. PE 取腾讯实时 TTM 而非 baostock 日线
- 腾讯盘中实时 PE(TTM)，baostock T+1 日线收盘 PE。腾讯 > baostock 降级链。

### 3. 市值取腾讯在线值，不再依赖手维种子数据
- 在线不可用时直接返回空，不拿旧数据冒充。

### 4. 行业行情独立获取，不从个股推导
- 指数点位/涨跌幅走新浪指数接口，PE 中位数和总市值从成分股在线值汇总推导。

### 5. 成分股只保留 code + name，删除硬编码 PE/市值
- `_INDUSTRY_SKELETON` 原来附带 `pe` 和 `market_cap_billion`，已移除。
- baostock 也不再从骨架取 PE/市值兜底，改为返回 None。
- 逐字段降级逻辑中，PE 和市值的 fallback 改为 "无在线源"。

### 6. 全链路丢弃种子数据，诚实面对"无数据"
- `data_fetcher.py` `_SEED_DATA`：8 个行业的价格/PE/市值 → 已删除
- `financial_data.py` `_SEED_FINANCIALS`：8 家公司的财务估算 → 已删除
- `news_data.py` `_SEED_NEWS`：4 个行业的新闻摘要 → 已删除
- `data_fetcher.py`：顶部 docstring 标注废弃，`_get_seed_data()` 方法删除

### 7. 编排层接入 DataSourceManager
- `agent_workflow.py` 阶段 1 从 `IndustryDataFetcher` 切换到 `DataSourceManager.gather()`
- 阶段 4 不再调用不存在的 `fetch_financial_data_tickers()`，改为从 `context["financial_data"]` 读取

### 8. 行业扩展设计：加行业只改一个文件
- 新行业只需在 `_INDUSTRY_SKELETON` 增加 code+name 列表
- 如果要加 BK 代码映射，在 `_INDUSTRY_BK_MAP` 加一行
- 所有下游流程（行情采集、四维 gather、agent 编排、prompt 模板）全部用 `{{INDUSTRY}}` 占位符，和市场名无关

## 六、遇到的问题及解决方案

| # | 问题 | 原因 | 解决方案 |
|---|------|------|---------|
| 1 | 东方财富 push2 API 调不通 | 本地 Clash 代理封锁了 eastmoney.com 域名 | 设置 `no_proxy` 绕过；改用新浪+腾讯+baostock 作为主力源 |
| 2 | akshare 也调不通 | akshare 底层也调东方财富 push2 | 放弃 akshare，改用独立数据源 |
| 3 | pytdx 连不上 | 自定义 TCP 协议被代理拦截 | 放弃 pytdx |
| 4 | 腾讯板块行情接口报错 | 接口路径不对 | 改用新浪指数接口（sz399363 国证算力） |
| 5 | PE 全 null | 新浪不返回 PE | 腾讯排到降级链第一层（有 PE），新浪降到第二层 |
| 6 | 涨跌幅解析错误 | 新浪字段解析逻辑有误 | 修正为 (现价 - 昨收) / 昨收 × 100 |
| 7 | 市值全 null | 腾讯字段 [45] 没解析 | 加解析：`parts[45]` 总市值，`parts[44]` 流通市值兜底 |
| 8 | 在线市值被种子覆盖 | `merge_with_seed` 取种子值丢了在线值 | 改为在线优先；最终直接删除种子数据 |
| 9 | 行业指数映射被误删 | 清理时误删 `_INDUSTRY_INDEX_MAP` | 补回 |
| 10 | 韦尔股份→豪威集团 | 公司更名 | 保留腾讯返回的实时名称 |
| 11 | cache 被提交到 git | 每次运行产生 diff | `.gitignore` 排除 `tools/cache/*.json` |
| 12 | 新旧代码并存 | `agent_workflow.py` 还在用老 `data_fetcher.py` | 切换到 `DataSourceManager.gather()` |
| 13 | 种子数据与在线数据冲突 | 新旧文件 PE/市值数值不一致 | 全部种子数据删除 |
| 14 | 非 AI 算力行业数据全挂 | 骨架只有 AI 算力一个行业 | 扩展到 14 个行业，每个 15 只成分股 |
| 15 | 东方财富 datacenter API 超时 | 同样被 Clash 代理拦截 + reportName 参数过期 | **已解决**（2026-07-10）— 用 baostock 替代东方财富作为财务数据主力源 |
| 16 | 东方财富新闻搜索 API 返回空 | `search-api-web.eastmoney.com` 接口参数过时/返回空 | **已解决**（2026-07-10）— 用 AKShare `stock_news_em` 替代，取龙头公司新闻聚合 |
| 17 | Python 内设置 `no_proxy` 无效 | urllib 在进程启动时已读取代理配置 | **已解决**（2026-07-10）— 删除无效的 `os.environ.setdefault`，全链不依赖东方财富 |
| 18 | 研报种子数据硬编码 | 只有 AI 算力有 3 条预采样 | **已解决**（2026-07-10）— 删除硬编码研报，暂无免费在线源 |
| 19 | announcement 巨潮公告不可靠 | 依赖 POST 到 cninfo.com.cn | 工作正常 ✅ — 保持不动 |
